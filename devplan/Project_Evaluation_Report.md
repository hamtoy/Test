## 📋 프로젝트 개요

### 🎯 목표 및 비전

- Google Gemini AI 기반으로 **Q&A 응답을 평가하고 재작성**하는 워크플로우를 제공하여, 품질·일관성·비용 효율을 동시에 개선합니다.
- 로컬 실행(메뉴/CLI)과 선택적 인프라(Neo4j/Redis/Web)를 조합해 **확장 가능한 엔터프라이즈형 파이프라인**으로 운영 가능하게 설계합니다.
- 테스트/정적분석/보안 스캔을 CI에 결합해 **운영 전 품질 게이트**를 자동화합니다.

### 👥 대상 사용자 및 주요 사용 시나리오

- 대상 사용자: QA 콘텐츠 운영자, 고객지원/문서팀, 데이터 품질 담당자, AI 워크플로우 엔지니어
- 사용 시나리오:
  - OCR/원문 기반 질의 생성 → 후보 답변 평가 → 최적 답변 재작성/검수
  - 캐시 통계 분석을 통한 비용 절감 효과 추적
  - (옵션) Neo4j 기반 RAG로 도메인 지식/규칙을 검색·활용하여 답변 품질 강화
  - (옵션) FastAPI Web API로 서비스화 및 대시보드/클라이언트 연동

### 🧭 에코시스템 내 포지션

- LLM 호출(생성/평가)과 데이터 계층(RAG/캐시), 운영 계층(CI/관측성)을 하나의 워크플로우로 묶는 **품질 개선 오케스트레이션 레이어** 역할을 수행합니다.
- 기본 설치(핵심 워크플로우) + 옵션 설치(RAG/Web/Worker/Observability)로 **점진적 도입**이 가능합니다.

## 🧩 주요 모듈별 상세 평가

### 1) 실행/CLI 레이어 (`src/main.py`, `src/cli.py`, `src/ui/`)

- **기능 완성도:** 메뉴 기반 실행 흐름이 명확하며(대화형), 비대화형 옵션/체크포인트도 제공됩니다.
- **코드 품질:** 구조가 단순하고 테스트 친화적 import가 일부 포함됩니다(테스트에서 재사용).
- **에러 처리:** 초기화 실패 시 종료 코드/로그를 남기며 사용자 메시지를 제공합니다.
- **성능:** 주로 I/O/LLM 호출에 지배되며, 시작 시 환경/템플릿 검증으로 빠른 실패가 가능합니다.
- **강점:** 진입점이 명확하고 운영 옵션이 풍부합니다(`--resume`, `--analyze-cache` 등).
- **약점 / 리스크:** `src/cli.py`가 모듈 로드시 외부 의존(예: Neo4j 드라이버)에 결합될 소지가 있어 “기본 설치” 경험을 훼손할 수 있습니다.

### 2) 에이전트/워크플로우 코어 (`src/agent/`, `src/workflow/`, `src/features/`)

- **기능 완성도:** 평가→재작성→검수/편집, LATS/자기교정 등 핵심 워크플로우 구성요소가 갖춰져 있습니다.
- **코드 품질:** 패키지 분리가 잘 되어 있고, 설정/인터페이스 계층을 통해 결합도를 낮추려는 흔적이 확인됩니다.
- **에러 처리:** 재시도/레이트리밋/예산 관리 등 실패 대비 설계가 포함되어 있습니다.
- **성능:** 동시성/캐싱/리트라이 정책이 존재하며, 비용/토큰 추적을 통한 최적화 여지가 큽니다.
- **강점:** 테스트 자산이 풍부하고(단위/통합/E2E), 기능 플래그 기반으로 옵션 기능을 제어합니다.
- **약점 / 리스크:** 옵션 기능이 많아 “기본 경로”와 “옵션 경로”의 경계가 흐려질 경우 회귀 위험이 존재합니다.

### 3) LLM 통합 (`src/llm/`)

- **기능 완성도:** Gemini 클라이언트, 모델 목록 조회, 타입 정의, (옵션) 체인(LangChain/LCEL) 구성이 포함됩니다.
- **코드 품질:** 타입/모듈 경계를 명확히 두고, v3.0 공개 API를 `src/__init__.py`에서 lazy-load로 관리합니다.
- **에러 처리:** 외부 라이브러리/옵션 의존성에 대해 graceful degradation 패턴이 일부 적용되어 있습니다.
- **성능:** 체인 최적화 경로(LCEL)가 존재하며, 캐싱/레이트리밋과 결합해 비용 제어가 가능합니다.
- **강점:** CI에서 정적 분석과 커버리지를 강제하여 변경 안정성이 높습니다.
- **약점 / 리스크:** 운영 시 API 키/모델 설정 실패가 치명적이므로, 사용자 친화적 오류 메시지와 “오프라인/더미” 경로의 구분이 더 명확하면 좋습니다.

### 4) 데이터·RAG/그래프 (`src/qa/`, `src/graph/`, `src/processing/`)

- **기능 완성도:** Neo4j 기반 지식그래프 및 벡터 검색, 규칙/제약 로딩, 컨텍스트 증강 등 RAG 파이프라인이 구현되어 있습니다.
- **코드 품질:** 모듈 수가 많고 기능 범위가 넓어, 공통 유틸/데이터 변환 계층의 정리가 중요합니다.
- **에러 처리:** 연결 실패 시 기능 비활성화로 폴백하는 흐름이 존재하며(Web 초기화 로직에서 확인됩니다).
- **성능:** 외부 DB/벡터 검색 비용이 크므로 배치/캐싱/인덱스 최적화(Two-tier) 전략이 중요합니다.
- **강점:** 옵션 기능을 통해 지식 기반 품질 향상이 가능하며, CI에서 Neo4j 서비스까지 포함한 테스트가 준비되어 있습니다.
- **약점 / 리스크:** 유사 기능(예: Data2Neo 관련 모듈)이 패키지 간 중복될 경우 유지보수 비용이 증가합니다.

### 5) Web API (`src/web/`)

- **기능 완성도:** FastAPI 앱, 라우터 분리, 미들웨어(request-id/에러/성능), 정적 파일/템플릿 제공이 구현되어 있습니다.
- **코드 품질:** 서비스 레지스트리/의존성 주입으로 초기화 흐름을 통제하며, 테스트 호환을 위한 backward compatibility도 고려합니다.
- **에러 처리:** 예외를 구조화 로깅하고 500 응답에 request_id를 포함하는 등 운영 진단성이 좋습니다.
- **성능:** 느린 요청만 경고로 로깅하고 응답 헤더에 처리 시간을 포함합니다.
- **강점:** 헬스 체크/메트릭 라우터를 제공하여 운영 모니터링 기반을 갖춥니다.
- **약점 / 리스크:** CORS/운영 설정이 코드에 고정된 부분은 배포 환경별 설정화가 필요하며, 전역 상태(호환성 목적)는 장기적으로 정리 대상입니다.

### 6) 캐싱·관측성·인프라 (`src/caching/`, `src/monitoring/`, `src/infra/`)

- **기능 완성도:** 컨텍스트 캐싱, 캐시 통계, Prometheus 메트릭, 헬스 체크, OpenTelemetry 헬퍼 등이 제공됩니다.
- **코드 품질:** “옵션 의존성은 지연 초기화” 패턴이 일부 적용되어 런타임 안정성을 높입니다.
- **에러 처리:** 외부 의존성 미설치 시 기능을 조용히 비활성화하는 경로가 있으므로, 운영 시 가시성(로그/경고) 기준이 중요합니다.
- **성능:** `cache_stats.jsonl` 분석이 대용량일 경우 메모리/시간 비용이 증가할 수 있어 스트리밍 처리가 유리합니다.
- **강점:** CI에 성능/보안/품질 체크가 포함되어 품질 회귀를 구조적으로 억제합니다.
- **약점 / 리스크:** 분석/리포팅 계열 유틸이 늘어날수록 IO·포맷·경로 규칙의 표준화가 필요합니다.

### 7) 프론트엔드 및 E2E (`package.json`, `playwright.config.ts`, `vite.config.ts`)

- **기능 완성도:** Vite/Vitest/Playwright 구성이 존재하며, UI/대시보드 확장 여지가 있습니다.
- **코드 품질:** 별도 패키지로 분리되어 있으나, 백엔드 API와의 계약(스키마/클라이언트) 정합성은 추가 정리가 필요합니다.
- **에러 처리:** 프론트 단 테스트/빌드 파이프라인은 존재하나, 백엔드 변경과 연동된 회귀 방어선은 강화 여지가 있습니다.
- **성능:** 번들/정적 배포 관점의 최적화는 후속 과제로 남습니다.
- **강점:** 백엔드와 별개로 빠른 UI 실험/배포가 가능합니다.
- **약점 / 리스크:** 백/프론트 빌드·배포 동선과 단일 소스 오브 트루스(API 계약) 정리가 지연되면 운영 복잡도가 증가합니다.
